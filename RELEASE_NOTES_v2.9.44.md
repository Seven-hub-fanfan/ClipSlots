# ClipSlots v2.9.44

本版为「默认槽位组」残留问题的**根因修复版**。此前 v2.9.42 / v2.9.43 修的都是 CLI 层的表象，本版终于定位并修复了真正的 GUI 层根因，可放心升级。

## 🐛 修复

### 「默认槽位组」残留的真正根因：GUI 切换页面时的 `else` 补建分支

- **现象**：即便在 v2.9.43 把 `create-page --group-name` 改为单次原子写之后，某些场景下目标页面上仍会冒出一个空的「默认槽位组」，与用户指定名称的组并存。
- **根因**：真正的元凶不在 CLI，而在 GUI 的 `switchToPage`（`SpecialSlotStorage.swift`）。该函数的 `else` 分支会在「当前页面查不到任何槽位组」时**自动补建**一个名为「默认槽位组」的组。当 GUI 在 CLI 写盘刚落地前的极短窗口（或写盘后拿到过期快照）里调用 `switchToPage` 时，`groupsInPage` 会瞬时表现为空，于是该分支被触发，静默注入了第二个「默认槽位组」。这与 v2.9.33 之后 `createPage` 原子建组的逻辑直接冲突。
- **修复**：**删除 `switchToPage` 的 `else` backfill 分支**。不再在切页时自动补建「默认槽位组」。此改动是安全的：
  1. 自 v2.9.33 起 `createPage` 总是同步创建首个槽位组，正常流程下不存在「零槽位组」的页面；
  2. 即便遇到损坏数据导致页面确实没有组，让选中指针保持 `nil` 也远比注入一个幽灵组更安全，UI 能够优雅处理 `nil` 选中态。

## 📌 修复层次说明

- **v2.9.42 / v2.9.43**：修的是 **CLI 层**（`create-page --group-name` 两步写 → 单次原子写），消除了 CLI 侧的中间态。
- **v2.9.44**：修的是 **GUI 层的真正根因**（删除 `switchToPage` 的 `else` 自动补建分支），从源头杜绝「默认槽位组」被重新注入。

## 🔧 兼容性

- 对既有页面 / 组数据无迁移影响，可直接覆盖升级。
- CLI 与 GUI 共享磁盘数据，行为一致，无接口变更。
