# ClipSlots v2.9.43

本版为一个**并发一致性修复版**，聚焦 v2.9.42 引入的 `create-page --group-name` 在与 GUI 实时监听并发时可能残留一个「默认槽位组」的问题。修复后 `--group-name` 建页不再产生废组，可安全升级。

## 🐛 修复

### `create-page --group-name` 与 GUI 并发竞态导致「默认槽位组」残留

- **现象**：CLI 执行 `create-page <页面名> --group-name <组名>` 时，在 GUI 正在运行（磁盘监听/自动重载开启）的情况下，偶发在目标页面上同时出现一个用户指定名称的组**和**一个名为「默认槽位组」的空组，即本应被「改名」掉的默认组以原名残留了下来。
- **根因**：v2.9.42 的 `--group-name` 是按「两步写」实现的——先落盘一个名为「默认槽位组」的 `defaultGroup`，随后再发起一次 `renameSpecialSlot` 把它改名。这两次写盘之间存在时间窗口，GUI 的磁盘监听会在中间态读到「默认槽位组」并缓存/回写，导致后续改名无法覆盖 GUI 侧持有的旧快照，最终两个组并存。
- **修复**：将「建默认组 + 改名」合并为**单次原子写**。当传入 `--group-name` 时，页面的第一个槽位组在**创建那一刻**就以目标名称落盘，不再经过「默认槽位组」这一中间态，因此不存在可被 GUI 监听读到的中间窗口，竞态被彻底消除。

## 🔧 兼容性

- **不传 `--group-name` 时行为完全不变**：仍返回名为「默认槽位组」的 `defaultGroup`，返回值结构与旧版一致。
- 传 `--group-name` 时返回结构不变，`defaultGroup.name` 即用户传入的名称；对既有页面/组数据无迁移影响，可直接覆盖升级。
